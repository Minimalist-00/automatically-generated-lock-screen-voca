# Development Guidelines & Feedback Loop

このプロジェクトにおける開発ルール、コンポーネントやデザインの管理場所についてのガイドラインです。
**【重要: フィードバックループ】**
共通コンポーネントを追加・変更したり、色使いなどのデザインルールを変更した場合は、**必ずこのファイル（`AGENTS.md`）の該当箇所も更新してください**。常にこのドキュメントが最新のプロジェクトの現状を表すようにします。これにより、AIエージェントと開発者（ユーザー）双方が常に最新の仕様を把握し、重複開発や仕様の散逸を防ぐことができます。

## 1. プロジェクト構造と管理場所

- **共通UIコンポーネント**: `src/components/`
  - ボタンやカード、ナビゲーションなど、複数画面で使われるコンポーネントはここに配置します。
  - 新しく共通で使えそうなUIや機能を作る場合は、各ページのファイルに直接書くのではなく、ここにコンポーネントとして切り出してください。
  - `TTSButton.tsx` : 音声読み上げボタン。
  - `PasteButton.tsx` : クリップボードからのペーストを容易にし、完了時にチェックマークへと変わるマイクロインタラクション付きのボタン。
- **グローバルスタイル・デザインルール**: `src/app/globals.css`
  - 色の定義（CSS変数）、共通のボタンスタイル（`.cute-btn`など）、フォント設定などが一元管理されています。

## 2. デザインシステム・色使いのルール

### カラーパレット（`globals.css` 参照）
- `--background`: `#EAF5F2` (背景)
- `--foreground`: `#4A6B65` (テキストなど)
- `--primary`: `#92D0C6` (メインカラー)
- `--primary-hover`: `#7BC0B5`
- `--secondary`: `#D1EAE5`
- `--secondary-hover`: `#C0DFD9`

### 共通UIクラス
統一感のあるデザイン（丸みのある柔らかいデザイン）を保つため、ボタンやカードには以下のような `globals.css` で定義されたクラスを使用してください。個別のコンポーネントで直接色や影を指定することはできる限り避けてください。

- **ボタン**:
  - `.cute-btn` : メインアクション用
  - `.cute-btn-secondary` : サブアクション用
  - `.cute-btn-outline` : 枠線のみのボタン
- **カード/コンテナ**: `.cute-card`
- **入力フォーム**: `.cute-input`
- **チェックボックス**: `.cute-checkbox`
- **アイコン**: `Material Symbols Rounded` を使用（絵文字の直接使用は避ける）

## 3. 開発フロー
1. 新規機能や画面を作成する際は、既存の `src/components/` や `.cute-` クラス群を最大限再利用する。
2. 既存のスタイルやコンポーネントで対応できない場合は、`src/components/` に新しいコンポーネントを作るか、`src/app/globals.css` に新しい共通ルールを追加する。
3. **その際、必ずこのドキュメント（`AGENTS.md`）にその新しいルールやコンポーネントの存在を追記する。**
4. コードが散逸するのを防ぐため、一つの共通的な修正（例えばボタンの形や色）を行うときは、`globals.css` のみを修正することでアプリケーション全体に反映されるように設計する。

## 4. 画面構成と主な機能

現在のアプリケーションは以下の主要な画面と機能で構成されています。機能を拡張・修正する際は、機能がどの画面の責務かを意識して実装してください。

- **ホーム画面 (`/`)**
  - ロック画面用壁紙のプレビュー表示および生成。
  - スマホでの壁紙生成・ダウンロードを高速化するため、`html-to-image` ではWebフォント埋め込みを回避し、ダウンロードには Web Share API (`navigator.share`) を使用しています。フォントはシステムフォント（iOSならヒラギノ丸ゴなど）で代替しています。
- **単語管理画面 (`/words`)**
  - 学習したい英単語の一覧表示と管理。
  - 単語の個別追加、一括追加 (Bulk Add。候補選択時に不要な単語を削除可能)。
  - ドラッグ＆ドロップによる並び替え、アーカイブ、削除機能。
  - AI例文生成時に、設定のプロンプトを使用するか、自分で独自のプロンプトを入力して生成するかを選択できる機能。
- **壁紙管理画面 (`/wallpapers`)**
  - 生成された壁紙のギャラリー表示。
  - 壁紙の名称変更 (Edit) と削除 (Delete) などのCRUD操作。
- **設定画面 (`/settings`)**
  - アプリケーション全体の各種設定。
- **グローバル機能 (全画面共通)**
  - **クイック追加機能** (`QuickAddFAB.tsx`): どの画面からでも思いついた時に素早く単語を登録できるフローティングボタン。

## 5. データベースの更新・マイグレーションについて

- **直接接続の制限**:
  - セキュリティやネットワーク設定により、ローカル環境（`migrate.js` などのスクリプト経由）から Supabase データベースへの直接接続（PostgreSQL ポート 5432/6543）はタイムアウトになり失敗します。
- **データベース変更の手順**:
  - カラムの追加などデータベースのスキーマを変更する場合は、**ローカルから直接接続して実行するのではなく、必ずブラウザ上の Supabase ダッシュボード（SQL Editor）から手動で SQL を実行**してください。
  - 変更後は、構成の同期や履歴管理のために `supabase/schema.sql` および `migrate.js` にもスキーマ定義やクエリを記録として追記してください。

## 6. 型定義の管理場所 (`types/`)

- プロジェクト全体で利用するTypeScriptの型定義（`interface` や `type`）やSupabaseのスキーマ型は、各コンポーネントやファイル内にインラインで記述して散在させるのではなく、**必ず `types/` ディレクトリ配下に集約**してください。
- 各コンポーネントやAPIルートからは、この `types/` 配下のファイルから型をインポートして適用してください。

## 7. やってはいけないこと（アンチパターン）

このプロジェクトにおいて「過去に問題が起きた」「AIが間違えやすい」アンチパターンを以下に記載します。
**【重要】開発中に新たなアンチパターンや「やってはいけないこと」を発見した場合は、AIエージェント自身で必ずこのセクションに追記し、以後の開発で同じミスが起きないようにしてください。**

- **UIへの直接的なスタイル指定（色のハードコードなど）**
  - 各コンポーネント内で個別に色や影を指定せず、必ず `globals.css` で定義されたクラスや変数を使用すること。
- **ローカルからのSupabase直接変更**
  - DBのスキーマ変更はローカルのスクリプトから直接行わず、ブラウザ上のSupabaseダッシュボードから行うこと（タイムアウトエラーを防ぐため）。
- **壁紙生成時のWebフォント埋め込み**
  - モバイル環境での生成パフォーマンス低下を防ぐため、`html-to-image` の処理ではWebフォントを埋め込まず、システムフォントを使用すること。
- **型定義の分散**
  - 上記「6. 型定義の管理場所」の通り、型定義は各ファイルに直接書かず、`types/` 配下にまとめること。
- **ロック画面の時計エリアへの要素配置（被り防止）**
  - 生成する壁紙の上部（おおよそ上から30%までの領域）は、iOSのロック画面の時計や日付と被るため、目標（Goal）や単語カードを配置しないこと。要素は必ず時計の下（`top: 32%` 以降など）から配置すること。

## 8. 自動テストとテスト駆動開発 (TDD) の徹底ルール

このプロジェクトでは、品質を担保しバグの再発を防ぐため、AI（Antigravity）は**ユーザーからの明示的な指示がなくても**以下のルールを自動的に適用・実行してください。

1. **バグ修正時のテスト駆動開発 (TDD) の義務**
   - ユーザーからバグや問題点が報告された場合、直接修正コードを書くのではなく、**「まず、そのバグを再現する（現在落ちる）テストコードを書く」**ことから始めてください。
   - その後、そのテストがパスするように実装を修正してください。これにより、将来的なデグレ（再発）を永遠に防ぎます。
2. **新規機能・コンポーネント追加時の自動テスト作成**
   - 共通コンポーネントや重要なロジック（API、Contextなど）を新しく追加または大きく変更した際は、必ず関連する `*.test.ts` または `*.test.tsx` ファイルを同時に作成（更新）してください。
   - ユーザーが「テストも書いて」と指定しなくても、AIが自律的にクロスチェックを行い、必要なテストを網羅してください。
3. **継続的フィードバックループの実行**
   - コードの変更やテストの追加を行った後は、ユーザーに完了報告をする前に、必ず背後で `npm run test` （または該当ファイルのテスト）を実行し、テストがすべてパスしていることを確認してください。
   - テストが落ちた場合は、ユーザーに報告する前にAI自身で原因を調査し、修正と再テストを行ってください（自律的なフィードバックループ）。
